The Problem

The reproducibility crisis, and the core concepts behind fixing it

What is Reproducible Research?

The ability to recreate scientific findings using:

  • Original data
  • Original code and analysis scripts
  • Documentation of computational environment
  • Clear methodology and workflow

Reproducibility is fundamental to the scientific method

Why Reproducibility Matters

Credibility

  • Builds trust in findings
  • Enables verification

Efficiency

  • Saves time later
  • Facilitates updates

Collaboration

  • Others can build on work
  • Easier knowledge transfer

Transparency

  • Clear methods
  • Visible decisions

The Reproducibility Crisis

A 2016 Nature survey of 1,576 researchers found:

  • >70% failed to reproduce another scientist’s experiments
  • >50% failed to reproduce their own experiments
  • Selective reporting & pressure to publish cited as top causes

Warning

Irreproducible preclinical research alone is estimated to cost $28 billion/year in the US.

Baker, M. (2016). “1,500 scientists lift the lid on reproducibility.” Nature.

Reproducible vs. Replicable

Term Same data? Same code? Question answered
Reproducible ✅ ✅ Do I get the same result from the same analysis?
Replicable ❌ ✅ Does the finding hold with new data?
Robust ✅ ❌ Does it hold with a different method?
Generalizable ❌ ❌ Does it hold across data and methods?

Literate Programming

Code and narrative living in one document

Literate Programming

“Instead of instructing a computer what to do, let us concentrate on explaining to humans what we want the computer to do.” — Donald Knuth (1984)

The idea

  • Code and narrative live in one document
  • Prose explains the why; code shows the how
  • Output is generated, never copy-pasted

Why it matters

  • No stale results from manual updates
  • The document is the analysis
  • Powers R Markdown, Quarto, MyST & Jupyter

Quarto

Language-agnostic scientific publishing

What is Quarto?

Next-generation scientific publishing system

  • Successor to R Markdown
  • Language-agnostic: R, Python, Julia, Observable
  • Multiple output formats: HTML, PDF, Word, presentations
  • Built-in support for citations and cross-references
  • Developed by Posit

MyST Markdown

Interactive books, Jupyter, and mybinder

What is MyST Markdown?

Markedly Structured Text

  • Extension of CommonMark Markdown
  • Designed for scientific and technical documentation
  • Part of the Jupyter ecosystem (Jupyter Book, Sphinx)
  • Supports roles and directives from reStructuredText
  • Excellent for creating interactive books

MyST Markdown Syntax

Here is example MyST syntax:

# My Document

:::{note}
This is a special note block!
:::

Here's an equation: $E = mc^2$

:::{admonition} Important
:class: tip
Remember to document your work!
:::

MyST Features

Directives

Special blocks

Roles

Jupyter notebooks as source

What is mybinder?

mybinder.org: Turn Git repositories into interactive notebooks

  • Zero installation for users
  • Reproducible environments via configuration files
  • Share live code with anyone
  • Free and open source

Perfect combination: MyST Markdown + mybinder = Interactive, reproducible documentation

Why MyST + mybinder?

Traditional Documentation

  • Static PDFs or HTML
  • Code examples can’t be run
  • Users must set up environment
  • Version mismatches
  • Hard to experiment

MyST + mybinder

  • Interactive notebooks
  • Click to run in browser
  • Environment pre-configured
  • Always reproducible
  • Easy experimentation

Adding mybinder to Your Project

1. Add a badge to README.md:

[![Binder](https://mybinder.org/badge_logo.svg)](
  https://mybinder.org/v2/gh/username/repo/main)

2. Create requirements.txt or environment.yml

3. Push to GitHub

4. Click the badge - done!

Tip

mybinder automatically detects configuration files and builds your environment

When to Use MyST?

Choose MyST When:

✅ Building interactive books

✅ Jupyter-centric workflow

✅ Need mybinder integration

✅ Python-heavy analysis

✅ Want Sphinx ecosystem

✅ Course materials online

Real Use Cases:

  • Research documentation (UW-Madison example)
  • Course textbooks with live code
  • Data analysis reports with reproducible results
  • API documentation with examples
  • Tutorial websites with interactive exercises

MyST Best Practices

  1. Use Jupyter notebooks as source files
  2. Add mybinder badge for interactivity
  3. Document dependencies clearly (requirements.txt)
  4. Cache outputs for faster builds
  5. Use directives for better formatting
  6. Cross-reference figures and tables
  7. Version control everything (including data when possible)
  8. Test mybinder link before sharing

Reproducible Environments

renv, Docker, and archiving for the long run — plus best practices

Best Practices for Reproducibility

  1. Version Control: Use Git to track all changes
  2. Environment Management:
    • Python: requirements.txt, conda env export
    • R: renv::snapshot(), renv.lock
  3. Clear Documentation: Explain what, why, and how
  4. Modular Code: Separate data, analysis, visualization
  5. Data Provenance: Record sources and transformations
  6. Automated Workflows: Use targets, make, drake

library(ggplot2)
library(dplyr)
library(tidyr)

# Set seed for reproducibility
set.seed(42)

# Create sample dataset
df <- data.frame(
  Method = rep(c('Traditional', 'Reproducible'), each = 50),
  Time_Saved = c(rnorm(50, mean = 5, sd = 2),
                 rnorm(50, mean = 15, sd = 3)),
  Errors = c(rpois(50, lambda = 5),
             rpois(50, lambda = 2))
)

# Create visualization
library(patchwork)

p1 <- ggplot(df, aes(x = Method, y = Time_Saved, fill = Method)) +
  geom_boxplot(alpha = 0.7) +
  scale_fill_manual(values = c("#E69F00", "#56B4E9")) +
  labs(title = "Time Saved (hours/month)",
       x = NULL, y = "Hours") +
  theme_minimal() +
  theme(legend.position = "none")

p2 <- ggplot(df, aes(x = Method, y = Errors, fill = Method)) +
  geom_boxplot(alpha = 0.7) +
  scale_fill_manual(values = c("#E69F00", "#56B4E9")) +
  labs(title = "Number of Errors",

Using renv for Reproducibility 

# Initialize renv for your project
renv::init()

# Install packages
install.packages("ggplot2")
install.packages("dplyr")

# Take a snapshot of dependencies
renv::snapshot()

# Restore dependencies on another machine
renv::restore()

Best Practice

Always use renv to manage R package dependencies for reproducible research!

Containerization with Docker

Capture the entire computational environment — OS, system libraries, language, and packages — in a single portable image.

FROM rocker/r-ver:4.3.0        # pinned R version

RUN R -e "install.packages('renv')"
COPY renv.lock renv.lock        # restore exact package versions
RUN R -e "renv::restore()"

COPY . /project
WORKDIR /project
CMD ["Rscript", "analysis.R"]

Tip

renv pins packages; Docker pins everything else. Together they make an analysis reproducible years later, on any machine.

Archiving & Citable Research

Make it permanent

  • Zenodo / OSF mint a DOI for your repo
  • GitHub → Zenodo integration archives each release
  • A DOI makes code & data citable

FAIR principles

  • Findable — persistent identifiers
  • Accessible — open, documented
  • Interoperable — standard formats
  • Reusable — clear licensing

Note

GitHub is for collaboration; an archive with a DOI is for permanence.

Start Small: The Adoption Ladder

You don’t need everything at once — each rung is useful on its own and takes minutes.

  1. Write in Quarto/R Markdown — prose + one code chunk = literate programming (zero setup)
  2. renv::snapshot() — one command pins your packages so it runs the same next year
  3. requirements.txt + Binder badge — anyone runs your work in a browser, no install
  4. Docker or a Zenodo DOI — only when you need exact environments or citable archives

Tip

The value compounds, but so does the effort — stop at whatever rung fits your needs.

Key Takeaways

  1. Reproducibility is the foundation of credible science
  2. Literate programming keeps code and narrative in sync
  3. Quarto & MyST turn analyses into shareable, executable documents
  4. Pin your environment — renv, Docker, requirements.txt
  5. mybinder lets anyone run your work with zero setup
  6. Archive with a DOI to make research permanent and citable
